
## 二、面向下游业务的优化建议

### 优先级 P0（阻塞下游使用）

#### 2.1 Pipeline 输出贯穿 P0 全字段

**问题**：当前输出只有标准化映射（8列），缺少检验结果、患者上下文
**影响**：客户查询、指标对比、四象限分析全部无法使用

**改造方案**：

`pipeline.py` 的 `run()` 方法需要改造为 DataFrame 驱动（而非纯 name list 驱动）：

```python
# 改造后的核心流程：
def run(self, input_path: str, output_dir: str | None = None) -> dict[str, Any]:
    path = Path(input_path)
    df = self._load_input(path)                    # P0 完整 DataFrame
    df = self._apply_standardization(df)           # L1+L2 标准化结果合并到 df
    df = self._apply_ai_review(df)                 # AI 复核
    classified = self.reviewer.classify_dataframe(df)  # 按 confidence 分层
    self.reviewer.export_csv(classified, output_dir)
    return classified
```

**输出增强后的列**：


#### 2.2 参考范围结构化解析

**问题**：`reference_range_raw` 是原始文本（如 `"卵泡期:0.057-0.893；排卵期：0.121-12.0"`），下游无法直接用
**影响**：四象限分析需要知道正常值上下限，当前无法自动判断异常

**改造方案**：在 `result_parser.py` 中新增 `parse_reference_range()` 方法：

```python
def parse_reference_range(self, raw_ref: str) -> dict:
    """
    解析参考范围，返回:
    {
        'ref_min': float | None,       # 下限
        'ref_max': float | None,       # 上限
        'ref_text': str | None,        # 文本型参考值（如 "阴性"）
        'ref_conditions': list[dict],  # 多期/多条件参考值
        'is_simple_range': bool,       # 是否为简单数值范围
    }
    """
    # 处理模式：
    # "0.00-5.800"                        → min=0, max=5.8, simple=True
    # "<1.0"                              → min=None, max=1.0, simple=True
    # "5.0-8.0"                           → min=5.0, max=8.0, simple=True
    # " ~ "                               → 全空
    # "阴性（-）"                          → ref_text="阴性"
    # "卵泡期:0.057-0.893；排卵期：..."    → ref_conditions=[{phase, min, max}...]
```

**工作量**：新增方法 + 测试，约 1-2 天

#### 2.8 四象限分析数据准备

**场景**：按「异常程度 × 健康风险」将指标分四象限展示
**需求**：每个指标需要有异常偏离度和风险权重

**改造方案**：

1. 在 `standard_dict.csv` 中新增列：

```csv
risk_weight,risk_category
# risk_weight: 0.0-1.0，表示该指标异常时的健康风险权重
# risk_category: critical（关键）/ warning（关注）/ info（参考）
```

2. 新增异常偏离度计算：

```python
def calc_deviation(
    value: float,
    ref_min: float,
    ref_max: float,
) -> float:
    """
    计算偏离度（0=正常中心, 正数=偏高, 负数=偏低）
    归一化到 [-1, 1] 范围，超出参考范围更大则 > 1 或 < -1
    """
    mid = (ref_min + ref_max) / 2
    half_range = (ref_max - ref_min) / 2
    if half_range == 0:
        return 0.0
    return (value - mid) / half_range
```

3. 四象限分类：

```
         高风险
    ┌───────┬───────┐
    │ 重点  │ 紧急  │
    │ 关注  │ 处理  │
    ├───────┼───────┤
    │ 正常  │ 轻度  │
    │ 范围  │ 异常  │
    └───────┴───────┘
         低风险
    正常     异常
```

## 三、代码层面优化建议

### 3.1 dict_manager.py 补充 docstring

当前 `dict_manager.py` 是唯一缺少完整 docstring 的核心模块。建议为 `_build_lookup_index()`、`lookup()` 补充文档。

### 3.2 result_parser.py 增加 ER 科室专项处理

当前解析器是通用的，但 ER 科室的 resultValue 格式特殊（`7.4kg  (7.7-9.4)   偏低`）。建议：
- 新增 `_parse_er_composite()` 专项方法
- 按 dept_code 路由到不同解析策略

### 3.3 日志覆盖补全

当前只有 P0Preprocessor 和 L2EmbeddingMatcher 有 logger。建议：
- L1RuleCleaner 增加清洗统计日志（各规则命中次数）
- Pipeline 增加各阶段耗时日志
- AI Review 增加 API 调用日志（请求/响应摘要、耗时）

### 3.4 性能优化

当前 `_standardize_names()` 是逐条循环清洗。大数据量时可优化：
- L1 批量清洗已有 `clean_batch()`（带 tqdm），Pipeline 应调用它
- L2 `search_batch()` 已实现，Pipeline 应对未命中项做批量向量搜索而非逐条

### 3.5 错误容忍度

当前单条数据解析失败会导致整个 pipeline 中断。建议：
- ResultParser.parse() 内部 try-except，失败返回空结果 + 日志告警
- L1 clean() 同理
- Pipeline 增加 `--strict / --lenient` 模式选项
